2. フォームリクエストの作成

sail artisan make:request StoreTodoRequest

StoreTodoRequest.php の中身

use Illuminate\Validation\Rule;

public function authorize(): bool
{
    return true;
}

public function rules(): array
{
    return [
        'title' => ['required', 'string', 'max:100'],
        'description' => ['nullable', 'string', 'max:65535'],
        'status' => ['required', Rule::in(['未着手', '進行中', '完了'])],
        'started_at' => ['nullable', 'date'],
        'completed_at' => ['nullable', 'date', 'after_or_equal:started_at'],
    ];
}

public function attributes(): array
{
    return [
        'title' => 'タイトル',
        'description' => '詳細',
        'status' => 'ステータス',
        'started_at' => '開始日',
        'completed_at' => '完了日',
    ];
}
ポイント:
  • authorize() は true にしないと403エラー
  • after_or_equal:started_at で完了日が開始日以降であることを保証
  • attributes() でエラー文言を日本語化

3. コントローラーの create / store

use App\Http\Requests\StoreTodoRequest;
use App\Models\Todo;

public function create()
{
    return view('todos.create');
}

public function store(StoreTodoRequest $request)
{
    Todo::create($request->validated());

    return redirect()->route('todos.index')
        ->with('success', 'ToDoを作成しました');
}

本日詰まったポイント

トラブル①: Class "APP\Http\Requests\StoreTodoRequest" does not exist

現象

新規作成しようとするとクラスが見つからないエラーが発生。

原因

コントローラーの use 文で名前空間の大文字小文字を誤っていた。

NG:
use APP\Http\Requests\StoreTodoRequest;
OK:
use App\Http\Requests\StoreTodoRequest;

解決方法

PHPの名前空間は大文字小文字を厳密に区別する。composer.json の autoload 定義では "App\\" となっているため、App が正しい。

教訓: 名前空間のタイポは class not found の頻出原因。特に App/APP/app の取り違えに注意。

トラブル②: ステータス選択肢の先頭に > が表示される

現象

セレクトボックスを開いた時、「> 未着手」のように先頭に > が表示されてしまう。

原因

<option> タグの中身（テキスト部分）に、属性として書くべき selected 判定の式を記述していた。

NG: 
<option value="{{ $status }}">
    {{ old('status') === $status ? 'selected' : '' }}
    {{ $status }}
</option>
OK: 
<option value="{{ $status }}" {{ old('status') === $status ? 'selected' : '' }}>
    {{ $status }}
</option>

図解: HTMLのタグ構造

<option value="未着手" selected>未着手</option> ├──────┬──────┤├──┤├──┤ │ │ │ │ 属性 属性の値 属性 ← ここに selected を書く （タグの > より前） ↓↓↓ <option ...> ここはタグの「中身」（画面に表示される文字） </option> ↑ ここに selected を書いても、ただの文字として表示されてしまう

解決方法

selected は <option> の属性なので、タグの > より前に書く。三項演算子の結果として selected 文字列が出力されれば、ブラウザが属性として解釈してくれる。

トラブル③: ステータスが「in Valid」エラーになる

現象

フォーム送信すると「ステータス must be one of the following types...」のバリデーションエラー。
dd($request->all()) で確認すると status => "未着手" と正常に送られているのに、なぜかバリデーションを通らない。

原因

in: ルールの区切り文字を 日本語の読点「、」で書いていた。

NG:
'status' => ['required', 'in:未着手、進行中、完了']

→ 「未着手、進行中、完了」という1つの文字列として扱われる

OK:
'status' => ['required', 'in:未着手,進行中,完了']

→ 3つの値として正しく区切られる

図解: in ルールのパース

ASCIIカンマ「,」の場合: 'in:未着手,進行中,完了' ↓ Laravel がカンマで分割 ['未着手', '進行中', '完了'] ← 3つの許可値として認識 ✅ 日本語の読点「、」の場合: 'in:未着手、進行中、完了' ↓ Laravel はカンマを探す（見つからない） ['未着手、進行中、完了'] ← 1つの許可値として認識 ❌ ↓ 送信値 "未着手" は一致しないのでバリデーションエラー

解決方法

区切り文字は必ず半角カンマ「,」を使う。ただし、このような区切り文字のミスを防ぐため、Rule::in([...]) の形式を使うのがより安全。

use Illuminate\Validation\Rule;

'status' => ['required', Rule::in(['未着手', '進行中', '完了'])],
教訓:
  • 日本語入力中に読点「、」とカンマ「,」を取り違えやすいので注意
  • 配列で渡す Rule::in([...]) なら区切り文字の問題が発生しない
  • バリデーションエラーが「値は正しいのに弾かれる」場合、ルール側の記述ミスを疑う

デバッグに使ったテクニック

FormRequestクラスの prepareForValidation() メソッドでバリデーション前の値をダンプ:

protected function prepareForValidation(): void
{
    dd($this->all());
}

コントローラーの store() に dd() を書いても、バリデーションエラーで store() まで到達しないので動作しない。その場合は FormRequest 側でダンプする。

  2. フォームに入力して「作成」ボタンをクリック
  3. 一覧画面にリダイレクトされ、作成したToDoが表示されればOK
